# 4.1.1. Componentes de UI

## 1. Introdução

O frontend do projeto organiza a interface em componentes pequenos, reutilizáveis e com responsabilidade bem definida. Essa estrutura está concentrada em `apps/frontend/src/components` e é consumida pelas páginas da aplicação para evitar duplicação de markup, comportamento e estilos.

A estratégia adotada no projeto favorece composição: componentes simples, como botões, campos de formulário, badges e barras de progresso, são combinados para formar blocos maiores, como cards, cabeçalhos, trilhas e páginas de conteúdo. Isso facilita manutenção, padronização visual e evolução incremental da interface.

## 2. Objetivo

Os componentes de UI do projeto foram criados para atender quatro objetivos principais:

- reduzir duplicação de código na camada de apresentação;
- manter consistência visual entre telas diferentes;
- simplificar a montagem de páginas a partir de blocos reutilizáveis;
- concentrar regras de exibição e variações de estilo em um único lugar.

## 3. Tecnologias utilizadas
 
Os componentes são escritos em **React** com **TypeScript** e estilizados com **CSS Modules** por arquivo. O projeto utiliza **Vite** como bundler e **CSS custom properties** (variáveis como `--cr-purple-900`, `--cr-green-100`) para manter consistência de tokens visuais entre componentes.

## 4. Organização dos componentes

Os componentes atuais podem ser entendidos em três grupos:

- componentes base, usados diretamente em diferentes telas;
- componentes compostos, que combinam outros componentes internos;
- componentes de estrutura de página, responsáveis por organizar trilhas, leitura e navegação.

### 4.1. Componentes base

Esses componentes são os mais genéricos do projeto e servem como blocos de construção para a interface.

| Componente      | Papel no projeto                                           | Reutilização principal          |
| --------------- | ---------------------------------------------------------- | ------------------------------- |
| `PrimaryButton` | Botão principal para ações de formulário e confirmação     | `LoginPage` e `CadastroPage`    |
| `FormInput`     | Campo de formulário simples com rótulo e tipo configurável | `LoginPage` e `CadastroPage`    |
| `Badge`         | Etiqueta visual para representar status ou contexto        | `ModuleCard` e `TrailHero`      |
| `Breadcrumbs`   | Navegação hierárquica para localização da página           | `LessonPage` e `TrailPage`      |
| `ProgressBar`   | Representação visual de progresso percentual               | `TrailHero` e `TrailStatusCard` |

O `PrimaryButton` é usado como CTA em fluxos curtos, enquanto o `FormInput` cobre os campos básicos de autenticação e cadastro. O `Badge` resolve estados visuais em diferentes contextos, e o `Breadcrumbs` mantém a navegação coerente entre páginas de trilha e estudo.

#### 4.1.1. PrimaryButton
 
O `PrimaryButton` encapsula o comportamento e estilo padrão do botão de ação principal da aplicação. Aceita qualquer atributo HTML nativo de `<button>` via spread de props (`ButtonHTMLAttributes`), além de `children` para o conteúdo do botão.
 
```tsx
// Props
type PrimaryButtonProps = {
    children: ReactNode;
} & ButtonHTMLAttributes<HTMLButtonElement>;
```
 
**Exemplo de uso:**
```tsx
<PrimaryButton type="submit">ENTRAR</PrimaryButton>
```
 
**Estilo base:** fundo `#fbc02d` (amarelo), `width: 160px`, `height: 58px`, `font-size: 28px`. A página que o consome pode sobrescrever via seletor `.recover-form .primary-button`, por exemplo.
 
---
 
#### 4.1.2. FormInput
 
O `FormInput` combina um `<label>` com um `<input>` configurável, expondo qualquer atributo nativo de `<input>` via spread (`InputHTMLAttributes`). O `id` do input é gerado automaticamente a partir de `name` caso não seja informado.
 
```tsx
// Props
type FormInputProps = {
    label: string;
} & InputHTMLAttributes<HTMLInputElement>;
```
 
**Exemplo de uso:**
```tsx
<FormInput label="E-mail" type="email" name="email" placeholder="Digite seu email" />
<FormInput label="Senha"  type="password" name="password" />
```
 
**Estilo base:** borda `2px solid var(--cr-purple-900)`, altura `42px`, fundo transparente.
 
---
 
#### 4.1.3. Badge
 
O `Badge` é uma etiqueta visual que comunica estado de forma padronizada. Possui quatro variantes pré-definidas, cada uma com cor e label padrão, que podem ser sobrescritos via props.
 
```tsx
// Variantes disponíveis
type BadgeVariant = "concluido" | "em-progresso" | "bloqueado" | "trilha";
 
// Labels padrão por variante
"concluido"    → "CONCLUÍDO"          // fundo verde
"em-progresso" → "EM PROGRESSO"       // fundo âmbar
"bloqueado"    → "BLOQUEADO"          // fundo cinza
"trilha"       → "TRILHA DE APRENDIZADO" // fundo branco translúcido (sobre fundo escuro)
```
 
**Exemplo de uso:**
```tsx
<Badge variant="concluido" />
<Badge variant="em-progresso" label="NÃO INICIADO" />
<Badge variant="trilha" label="TRILHA DE REQUISITOS" />
```
 
---
 
#### 4.1.4. Breadcrumbs
 
O `Breadcrumbs` renderiza uma lista de itens de navegação hierárquica. O último item é tratado automaticamente como a página atual (`aria-current="page"`). Itens com `onClick` se tornam botões clicáveis; os demais são spans estáticos.
 
```tsx
// Props
interface BreadcrumbItem {
  label: string;
  onClick?: () => void;
}
```
 
**Exemplo de uso:**
```tsx
<Breadcrumbs items={[
  { label: "Trilhas", onClick: () => navigate("/trilhas") },
  { label: "Engenharia de Requisitos", onClick: () => navigate("/trilha/1") },
  { label: "Elicitação de Requisitos" },
]} />
```
 
---
 
#### 4.1.5. ProgressBar
 
O `ProgressBar` é uma barra de progresso configurável com suporte a dois tamanhos (`sm` / `md`), duas variantes de cor (`purple` / `green`), cabeçalho com label e percentual, e animação de entrada.
 
```tsx
// Principais props
interface ProgressBarProps {
  value: number;           // 0–100
  size?:           "sm" | "md";
  color?:          "purple" | "green";
  label?:          string;
  showPercentage?: boolean;
  animated?:       boolean;
}
```
 
**Exemplo de uso:**
```tsx

<ProgressBar value={42} size="md" color="green" label="42% CONCLUÍDO" animated />
 
// Barra lateral de módulo — cor roxa, tamanho pequeno
<ProgressBar value={75} size="sm" color="purple" showPercentage />
```
 
A classe `.cr-progress--on-dark` pode ser aplicada no elemento pai para adaptar as cores ao fundo escuro do `TrailHero`.
 
### 4.2. Componentes compostos

Esses componentes são formados pela combinação de componentes base, o que reduz complexidade nas páginas e padroniza a experiência.

| Componente        | Composição interna                                     | Reuso observado                               |
| ----------------- | ------------------------------------------------------ | --------------------------------------------- |
| `Header`          | Estrutura de topo com ações e navegação por ícones     | `AvailableTracks`, `LessonPage` e `TrailPage` |
| `TrailCard`       | Card com imagem, texto, progresso e ação de exploração | `AvailableTracks`                             |
| `TrailHero`       | Banner principal da trilha com `Badge` e `ProgressBar` | `TrailPage`                                   |
| `TrailStatusCard` | Painel lateral com lista de progresso por módulo       | `TrailPage`                                   |
| `ModuleCard`      | Card de módulo com ícone, `Badge` e botão de ação      | `TrailPage`                                   |
| `QuickTipCard`    | Card curto de destaque para dica rápida                | `TrailPage`                                   |

O `Header` funciona como identidade persistente da interface. Já `TrailCard`, `TrailHero`, `TrailStatusCard` e `ModuleCard` são variações de cards orientadas à trilha, com foco em status, progresso e ação do usuário.


#### 4.2.1. Header
 
O `Header` é o cabeçalho persistente da aplicação. Ele integra lógica de autenticação via hook `useAuth` e navegação via `useNavigate` do React Router. Renderiza condicionalmente os botões de módulos e logout apenas quando o usuário está autenticado.
 
```tsx
// Props
type HeaderProps = {
  onModulesClick?: () => void; // callback opcional para override de navegação
};
```
 
Os ícones (usuário, livro, logout) são SVGs inline, sem dependência de biblioteca de ícones externa.
 
---
 
#### 4.2.2. TrailCard
 
O `TrailCard` exibe uma trilha disponível com imagem de capa, título, descrição, percentual de progresso e botão de exploração.
 
```tsx
interface TrailCardProps {
  title: string;
  description: string;
  image: string;       // URL da imagem de capa
  progress: number;    // 0–100
  onExplore?: () => void;
}
```
 
---
 
#### 4.2.3. TrailHero
 
O `TrailHero` é o banner principal da página de trilha. Compõe `Badge` (variante `"trilha"`) e `ProgressBar` (cor `green`, tamanho `md`) sobre um fundo escuro com uma ilustração SVG de livro aberto gerada inline, sem imagem externa.
 
```tsx
interface TrailHeroProps {
  title: string;
  description: string;
  progress: number;      // 0–100
  badgeLabel?: string;   // sobrescreve o label padrão do Badge
}
```
 
---
 
#### 4.2.4. ModuleCard
 
O `ModuleCard` é o componente mais complexo da camada de componentes. Gerencia quatro estados de módulo (`LOCKED`, `NOT_STARTED`, `IN_PROGRESS`, `COMPLETED`), cada um com ícone SVG, variante de `Badge` e configuração de botão próprios.
 
```tsx
export type ModuleStatus = "LOCKED" | "NOT_STARTED" | "IN_PROGRESS" | "COMPLETED";
 
interface ModuleCardProps {
  number: number;
  title: string;
  description: string;
  status: ModuleStatus;
  onAction?: () => void;
}
```

 
#### 4.2.5. QuickTipCard
 
O `QuickTipCard` é um card simples de destaque para dicas rápidas, com fundo roxo e texto em citação. Aceita `text` e `label` customizável (padrão: `"DICA RÁPIDA"`).
 
```tsx
// Props
interface QuickTipCardProps {
  text: string;
  label?: string; // padrão: "DICA RÁPIDA"
}
```
 
**Exemplo de uso:**
```tsx
<QuickTipCard text="Leia os requisitos com atenção antes de validar com o cliente." />
<QuickTipCard label="SABIA QUE?" text="A elicitação de requisitos pode consumir até 30% do projeto." />
```
 
---
 

### 4.3. Componentes de apoio à leitura e estudo

Esses componentes organizam a experiência em páginas de conteúdo mais longas.

| Componente       | Papel no fluxo                                           | Reuso observado |
| ---------------- | -------------------------------------------------------- | --------------- |
| `ContentPreview` | Bloco para incorporar mídia de apoio                     | `LessonPage`    |
| `LessonSection`  | Seção de leitura com parágrafos, lista e ação de leitura | `LessonPage`    |
| `ReadingSidebar` | Barra lateral com etapas e progresso da leitura          | `LessonPage`    |

Esses componentes ajudam a separar conteúdo didático de navegação e acompanhamento, permitindo que a página de aula seja montada por partes sem repetir estrutura de layout.

#### 4.3.1. ContentPreview
 
O `ContentPreview` embute um player de vídeo do YouTube via `<iframe>` com `loading="lazy"` e política de referência restrita (`strict-origin-when-cross-origin`), seguindo boas práticas de privacidade e performance.
 
#### 4.3.2. LessonSection
 
O `LessonSection` é uma seção de leitura que recebe parágrafos, bullets opcionais e um estado de leitura (`read`). Ao clicar em "Marcar como lido", dispara `onMarkRead`. Usa `forwardRef` para permitir scroll programático pela sidebar.
 
```tsx
type LessonSectionProps = {
  id: string;
  title: string;
  paragraphs: readonly string[];
  bullets?: readonly string[];
  read: boolean;
  onMarkRead: () => void;
  children?: ReactNode; // slot para mídia de apoio
};
```
 
#### 4.3.3. ReadingSidebar
 
O `ReadingSidebar` exibe a lista de seções da aula com indicadores de estado (`pending`, `current`, `done`, `wrong`) e uma barra de progresso da leitura.
 
```tsx
type ReadingSidebarProps = {
  progress: number;
  progressLabel: string;
  items: Array<{
    label: string;
    state: "pending" | "current" | "done" | "wrong";
  }>;
};
```

## 5. Como a reutilização funciona no projeto

A reutilização no frontend segue um padrão simples e eficiente:

- componentes pequenos concentram uma única responsabilidade;
- componentes compostos reutilizam os componentes base em vez de reimplementar comportamento;
- a customização é feita por props, evitando forks de UI para cada tela;
- cada componente mantém seu próprio arquivo de estilo, o que facilita evolução sem espalhar regras pelo projeto.

Na prática, isso permite que o mesmo bloco visual assuma papéis diferentes conforme a tela. Por exemplo, `ProgressBar` representa progresso geral na trilha e também o progresso lateral dos módulos; `Badge` comunica status em cards diferentes; `Breadcrumbs` mantém a orientação do usuário tanto em leitura quanto em navegação de trilha.

## 6. Relação com as páginas da aplicação

Os principais pontos de consumo desses componentes no frontend atual são:

- `LoginPage` e `CadastroPage` usam `FormInput` e `PrimaryButton` para os fluxos de autenticação;
- `AvailableTracks` usa `Header` e `TrailCard` para apresentar trilhas disponíveis;
- `TrailPage` combina `Header`, `Breadcrumbs`, `TrailHero`, `TrailStatusCard`, `ModuleCard` e `QuickTipCard` para estruturar a visão da trilha;
- `LessonPage` combina `Header`, `Breadcrumbs`, `ContentPreview`, `LessonSection` e `ReadingSidebar` para organizar leitura e acompanhamento do progresso.

Esse arranjo mostra que os componentes não existem isoladamente. Eles foram pensados para compor páginas inteiras com uma linguagem visual consistente e com comportamento reaproveitável.

## 7. Visão crítica

A principal qualidade dessa base de componentes é a simplicidade de composição. Em vez de criar páginas monolíticas, o projeto distribui a interface em peças menores que podem ser combinadas e mantidas separadamente. Isso reduz acoplamento e torna mais fácil alterar uma tela sem quebrar as demais.

Outro ponto positivo é a padronização de padrões visuais recorrentes, como botões, badges, progresso e cartões de conteúdo. Mesmo em componentes com propósitos diferentes, a aplicação preserva uma leitura visual coerente.

Como ponto de atenção, alguns componentes ainda são bastante específicos de determinadas telas. Isso é aceitável no estágio atual, mas vale observar continuamente se algum bloco pode ser generalizado para evitar duplicação futura.

## 8. Conclusão

Os componentes de UI do projeto formam a base da reutilização no frontend. Eles cobrem desde elementos simples de formulário até estruturas compostas de navegação, progresso e leitura. Essa organização torna o código mais previsível, facilita a manutenção e permite que novas telas sejam montadas com menor esforço, reaproveitando blocos já consolidados.
