# 4.1.1. Componentes de UI

## 1. Introdução

O frontend do projeto organiza a interface em componentes pequenos, reutilizáveis e com responsabilidade bem definida. Essa estrutura está concentrada em `apps/frontend/src/components` e é consumida pelas páginas da aplicação para evitar duplicação de markup, comportamento e estilos.

A estratégia adotada no projeto favorece composição: componentes simples, como botões, campos de formulário, badges e barras de progresso, são combinados para formar blocos maiores, como cards, cabeçalhos, trilhas e páginas de conteúdo. Isso facilita manutenção, padronização visual e evolução incremental da interface.

## 2. Objetivo

Os componentes de UI do projeto foram criados para atender quatro objetivos principais:

- reduzir duplicação de código na camada de apresentação;
- manter consistência visual entre telas diferentes;
- simplificar a montagem de páginas a partir de blocos reutilizáveis;
- concentrar regras de exibição e variações de estilo em um único lugar.

## 3. Organização dos componentes

Os componentes atuais podem ser entendidos em três grupos:

- componentes base, usados diretamente em diferentes telas;
- componentes compostos, que combinam outros componentes internos;
- componentes de estrutura de página, responsáveis por organizar trilhas, leitura e navegação.

### 3.1. Componentes base

Esses componentes são os mais genéricos do projeto e servem como blocos de construção para a interface.

| Componente      | Papel no projeto                                           | Reutilização principal          |
| --------------- | ---------------------------------------------------------- | ------------------------------- |
| `PrimaryButton` | Botão principal para ações de formulário e confirmação     | `LoginPage` e `CadastroPage`    |
| `FormInput`     | Campo de formulário simples com rótulo e tipo configurável | `LoginPage` e `CadastroPage`    |
| `Badge`         | Etiqueta visual para representar status ou contexto        | `ModuleCard` e `TrailHero`      |
| `Breadcrumbs`   | Navegação hierárquica para localização da página           | `LessonPage` e `TrailPage`      |
| `ProgressBar`   | Representação visual de progresso percentual               | `TrailHero` e `TrailStatusCard` |

O `PrimaryButton` é usado como CTA em fluxos curtos, enquanto o `FormInput` cobre os campos básicos de autenticação e cadastro. O `Badge` resolve estados visuais em diferentes contextos, e o `Breadcrumbs` mantém a navegação coerente entre páginas de trilha e estudo.

### 3.2. Componentes compostos

Esses componentes são formados pela combinação de componentes base, o que reduz complexidade nas páginas e padroniza a experiência.

| Componente        | Composição interna                                     | Reuso observado                               |
| ----------------- | ------------------------------------------------------ | --------------------------------------------- |
| `Header`          | Estrutura de topo com ações e navegação por ícones     | `AvailableTracks`, `LessonPage` e `TrailPage` |
| `TrailCard`       | Card com imagem, texto, progresso e ação de exploração | `AvailableTracks`                             |
| `TrailHero`       | Banner principal da trilha com `Badge` e `ProgressBar` | `TrailPage`                                   |
| `TrailStatusCard` | Painel lateral com lista de progresso por módulo       | `TrailPage`                                   |
| `ModuleCard`      | Card de módulo com ícone, `Badge` e botão de ação      | `TrailPage`                                   |
| `QuickTipCard`    | Card curto de destaque para dica rápida                | `TrailPage`                                   |

O `Header` funciona como identidade persistente da interface. Já `TrailCard`, `TrailHero`, `TrailStatusCard` e `ModuleCard` são variações de cards orientadas à trilha, com foco em status, progresso e ação do usuário.

### 3.3. Componentes de apoio à leitura e estudo

Esses componentes organizam a experiência em páginas de conteúdo mais longas.

| Componente       | Papel no fluxo                                           | Reuso observado |
| ---------------- | -------------------------------------------------------- | --------------- |
| `ContentPreview` | Bloco para incorporar mídia de apoio                     | `LessonPage`    |
| `LessonSection`  | Seção de leitura com parágrafos, lista e ação de leitura | `LessonPage`    |
| `ReadingSidebar` | Barra lateral com etapas e progresso da leitura          | `LessonPage`    |

Esses componentes ajudam a separar conteúdo didático de navegação e acompanhamento, permitindo que a página de aula seja montada por partes sem repetir estrutura de layout.

## 4. Como a reutilização funciona no projeto

A reutilização no frontend segue um padrão simples e eficiente:

- componentes pequenos concentram uma única responsabilidade;
- componentes compostos reutilizam os componentes base em vez de reimplementar comportamento;
- a customização é feita por props, evitando forks de UI para cada tela;
- cada componente mantém seu próprio arquivo de estilo, o que facilita evolução sem espalhar regras pelo projeto.

Na prática, isso permite que o mesmo bloco visual assuma papéis diferentes conforme a tela. Por exemplo, `ProgressBar` representa progresso geral na trilha e também o progresso lateral dos módulos; `Badge` comunica status em cards diferentes; `Breadcrumbs` mantém a orientação do usuário tanto em leitura quanto em navegação de trilha.

## 5. Relação com as páginas da aplicação

Os principais pontos de consumo desses componentes no frontend atual são:

- `LoginPage` e `CadastroPage` usam `FormInput` e `PrimaryButton` para os fluxos de autenticação;
- `AvailableTracks` usa `Header` e `TrailCard` para apresentar trilhas disponíveis;
- `TrailPage` combina `Header`, `Breadcrumbs`, `TrailHero`, `TrailStatusCard`, `ModuleCard` e `QuickTipCard` para estruturar a visão da trilha;
- `LessonPage` combina `Header`, `Breadcrumbs`, `ContentPreview`, `LessonSection` e `ReadingSidebar` para organizar leitura e acompanhamento do progresso.

Esse arranjo mostra que os componentes não existem isoladamente. Eles foram pensados para compor páginas inteiras com uma linguagem visual consistente e com comportamento reaproveitável.

## 6. Visão crítica

A principal qualidade dessa base de componentes é a simplicidade de composição. Em vez de criar páginas monolíticas, o projeto distribui a interface em peças menores que podem ser combinadas e mantidas separadamente. Isso reduz acoplamento e torna mais fácil alterar uma tela sem quebrar as demais.

Outro ponto positivo é a padronização de padrões visuais recorrentes, como botões, badges, progresso e cartões de conteúdo. Mesmo em componentes com propósitos diferentes, a aplicação preserva uma leitura visual coerente.

Como ponto de atenção, alguns componentes ainda são bastante específicos de determinadas telas. Isso é aceitável no estágio atual, mas vale observar continuamente se algum bloco pode ser generalizado para evitar duplicação futura.

## 7. Conclusão

Os componentes de UI do projeto formam a base da reutilização no frontend. Eles cobrem desde elementos simples de formulário até estruturas compostas de navegação, progresso e leitura. Essa organização torna o código mais previsível, facilita a manutenção e permite que novas telas sejam montadas com menor esforço, reaproveitando blocos já consolidados.
