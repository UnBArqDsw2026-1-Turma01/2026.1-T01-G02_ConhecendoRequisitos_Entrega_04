# 4.4. Reutilização de módulos compartilhados entre as camadas Frontend e Backend

## 1. Introdução

O projeto **G02 — ConhecendoRequisitos** é organizado como um repositório com duas aplicações independentes em `apps/`: `apps/frontend` (React + Vite + TypeScript) e `apps/backend` (NestJS + Prisma). Diferente de um monorepo com Turborepo e pacotes publicáveis em `packages/`, a reutilização aqui acontece **dentro de cada camada**, por meio de módulos centrais que concentram uma responsabilidade específica e são consumidos por múltiplos pontos da aplicação, evitando duplicação de código.

Neste documento, são descritos os principais módulos reutilizáveis identificados no projeto:

- `api.ts`: instância única do Axios (com interceptors) usada por todos os serviços HTTP do frontend.
- `validators.ts`: funções utilitárias de validação de e-mail e senha, reutilizadas em diferentes páginas/formulários.
- `useAuth.ts`: hook que encapsula o `authStore` (Zustand), reutilizado por componentes de rota protegida, cabeçalho e páginas de autenticação.
- `auth.exceptions.ts`: conjunto de exceções customizadas do backend, reutilizadas em todos os fluxos do `AuthService` (registro, login, recuperação e redefinição de senha).

## 2. Propósito do Componente

O objetivo deste documento é registrar o papel dos **módulos de uso compartilhado** identificados no projeto, explicando:

- **O que eles fazem**:
  - Centralizar a configuração de comunicação HTTP com o backend, incluindo injeção automática do token JWT e tratamento de sessão expirada (`api.ts`).
  - Padronizar regras de validação de e-mail e força de senha usadas em formulários (`validators.ts`).
  - Expor de forma única o estado de autenticação (usuário, token, status) para qualquer componente da árvore React (`useAuth.ts`).
  - Padronizar os erros de domínio da autenticação no backend, com mensagens e códigos HTTP consistentes (`auth.exceptions.ts`).

- **Como foram estruturados**:
  - `api.ts` como uma **instância única do Axios**, criada uma vez e importada por todos os serviços (`auth.service.ts`, `trails.service.ts`).
  - `validators.ts` como um **objeto utilitário** com funções puras, sem dependências externas, fácil de testar e reaproveitar.
  - `useAuth.ts` como um **hook customizado** que envolve a store global (`useAuthStore`), evitando que cada componente acesse a store diretamente.
  - `auth.exceptions.ts` como um conjunto de **classes que estendem exceções nativas do NestJS** (`BadRequestException`, `ConflictException`, `UnauthorizedException`), lançadas pelo `AuthService`.

- **Como isso se conecta à arquitetura**:
  - Esses módulos reduzem duplicação de código dentro de cada camada (frontend e backend) e garantem consistência de comportamento (mesma forma de autenticar requisições, mesma forma de validar dados, mesma forma de sinalizar erros), mesmo sem um pacote compartilhado entre as duas camadas.

## 3. Reutilização do Componente

### 3.1. Módulo `api.ts` (camada Frontend)

Este módulo concentra a configuração da comunicação HTTP entre o frontend e o backend, sendo o ponto único de criação da instância do Axios.

#### 3.1.1. `apps/frontend/src/services/api.ts`

```ts
import axios, { AxiosError } from "axios";
import type { AxiosInstance, AxiosResponse } from "axios";

const API_BASE_URL = import.meta.env.VITE_API_URL || "http://localhost:3000";

// Criar instância axios
const api: AxiosInstance = axios.create({
  baseURL: API_BASE_URL,
  headers: {
    "Content-Type": "application/json",
  },
});

// Interceptor para adicionar token JWT em todas as requisições
api.interceptors.request.use(
  (config) => {
    const token = localStorage.getItem("auth_token");
    if (token) {
      config.headers.Authorization = `Bearer ${token}`;
    }
    return config;
  },
  (error) => Promise.reject(error),
);

// Interceptor para tratar erros
api.interceptors.response.use(
  (response: AxiosResponse) => response,
  (error: AxiosError) => {
    // Se token expirou (401), limpar localStorage e redirecionar para login
    if (error.response?.status === 401) {
      localStorage.removeItem("auth_token");
      localStorage.removeItem("auth_user");
      // Redirecionar será feito no componente
      window.location.href = "/login";
    }

    return Promise.reject(error);
  },
);

export default api;
```

**Reutilização:**

- Importado por `auth.service.ts`, que o usa para chamar `/auth/register`, `/auth/login`, `/auth/logout`, `/auth/forgot-password` e `/auth/reset-password`.
- Importado por `trails.service.ts`, que o usa em mais de uma dezena de chamadas (trilhas, módulos, conteúdos, quizzes e progresso).
- Qualquer novo serviço HTTP criado no frontend pode importar `api` em vez de instanciar um novo cliente Axios, herdando automaticamente o token JWT e o tratamento de expiração de sessão.

### 3.2. Módulo `validators.ts` (camada Frontend)

Funções utilitárias de validação, reaproveitadas em diferentes formulários sem repetir expressões regulares ou regras de negócio.

#### 3.2.1. `apps/frontend/src/utils/validators.ts`

```ts
export const validators = {
  isValidEmail: (email: string): boolean => {
    const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
    return emailRegex.test(email);
  },

  isStrongPassword: (password: string): boolean => {
    // Mínimo 8 caracteres, pelo menos uma letra maiúscula, uma minúscula e um número
    const passwordRegex = /^(?=.*[a-z])(?=.*[A-Z])(?=.*\d).{8,}$/;
    return passwordRegex.test(password);
  },

  isWeakPassword: (password: string): boolean => {
    // Mínimo 8 caracteres (mais fraco que strong)
    return password.length >= 8;
  },

  getPasswordStrength: (password: string): "fraca" | "média" | "forte" => {
    if (!password) return "fraca";

    let strength = 0;
    if (password.length >= 8) strength++;
    if (password.length >= 12) strength++;
    if (/[a-z]/.test(password) && /[A-Z]/.test(password)) strength++;
    if (/\d/.test(password)) strength++;
    if (/[^a-zA-Z\d]/.test(password)) strength++;

    if (strength <= 2) return "fraca";
    if (strength <= 3) return "média";
    return "forte";
  },
};
```

**Reutilização:**

- `CadastroPage.tsx`: usa `validators.isValidEmail` e `validators.isWeakPassword` na validação dos campos do formulário de cadastro antes de chamar `register_user`.
- `RecuperarSenhaPage.tsx`: reaproveita as mesmas funções para validar e-mail/senha no fluxo de recuperação de senha.
- Qualquer página futura que precise validar e-mail ou senha pode importar `validators` em vez de recriar as expressões regulares.

### 3.3. Hook `useAuth.ts` (camada Frontend)

Hook que encapsula o acesso à store global de autenticação (Zustand), centralizando a forma como os componentes leem e modificam o estado de sessão do usuário.

#### 3.3.1. `apps/frontend/src/hooks/useAuth.ts`

```ts
import { useAuthStore } from "../stores/authStore";

export const useAuth = () => {
  const {
    user,
    token,
    isAuthenticated,
    isLoading,
    error,
    setUser,
    setToken,
    setLoading,
    setError,
    logout,
    login,
    restoreSession,
  } = useAuthStore();

  return {
    user,
    token,
    isAuthenticated,
    isLoading,
    error,
    setUser,
    setToken,
    setLoading,
    setError,
    logout,
    login,
    restoreSession,
  };
};
```

**Reutilização:**

- `ProtectedRoute.tsx`: usa `isAuthenticated` para decidir se redireciona o usuário para `/login` ou renderiza a rota protegida.
- `Header.tsx`: usa o hook para exibir dados do usuário logado e oferecer a ação de `logout`.
- `LoginPage.tsx` e `CadastroPage.tsx`: usam `login`/`setError`/`setLoading` para refletir o resultado das chamadas de autenticação na interface.
- `main.tsx`: usa `restoreSession` para restaurar a sessão a partir do `localStorage` na inicialização da aplicação.

### 3.4. Módulo `auth.exceptions.ts` (camada Backend)

Conjunto de exceções de domínio que padronizam os erros lançados pelo fluxo de autenticação no backend.

#### 3.4.1. `apps/backend/src/common/exceptions/auth.exceptions.ts`

```ts
import {
  BadRequestException,
  ConflictException,
  UnauthorizedException,
} from "@nestjs/common";

export class EmailAlreadyExistsException extends ConflictException {
  constructor(email: string) {
    super(`Email ${email} já está cadastrado`);
  }
}

export class InvalidEmailException extends BadRequestException {
  constructor() {
    super("Email inválido");
  }
}

export class WeakPasswordException extends BadRequestException {
  constructor() {
    super("Senha deve conter pelo menos 8 caracteres");
  }
}

export class InvalidCredentialsException extends UnauthorizedException {
  constructor() {
    super("Email ou senha inválidos");
  }
}

export class InvalidTokenException extends UnauthorizedException {
  constructor() {
    super("Token inválido ou expirado");
  }
}

export class UserNotFoundException extends UnauthorizedException {
  constructor() {
    super("Usuário não encontrado");
  }
}
```

**Reutilização:**

- `AuthService.register`: lança `EmailAlreadyExistsException`, `WeakPasswordException` e `UserNotFoundException`.
- `AuthService.login`: lança `InvalidCredentialsException` tanto quando o usuário não existe quanto quando a senha não confere, evitando vazar qual das duas condições falhou.
- `AuthService.resetPassword`: lança `InvalidTokenException` (token inexistente ou expirado) e `WeakPasswordException` (nova senha fraca).
- Como todas estendem exceções nativas do NestJS, o `ExceptionFilter` padrão do framework já sabe transformá-las na resposta HTTP correta, sem necessidade de tratamento adicional em cada controller.

## 4. Utilização na Aplicação

### 4.1. `api.ts`

**Fluxo de Autenticação (frontend)**

- `auth.service.ts` importa `api` e o utiliza para `register`, `login`, `logout`, `forgotPassword`, `resetPassword` e `getMe`, sempre delegando a montagem da URL base e dos headers à instância compartilhada.

**Fluxo de Trilhas e Progresso (frontend)**

- `trails.service.ts` reutiliza a mesma instância `api` em mais de quinze chamadas distintas (trilhas, módulos, conteúdos, quizzes, progresso de aula/exercício/módulo/trilha), garantindo que todas elas se beneficiem do interceptor de token JWT e do redirecionamento automático em caso de sessão expirada (401).

### 4.2. `validators.ts`

**Cadastro de usuário**

- `CadastroPage.tsx` chama `validators.isValidEmail` e `validators.isWeakPassword` antes de submeter os dados via `authService.register`.

**Recuperação de senha**

- `RecuperarSenhaPage.tsx` reutiliza as mesmas funções para validar o e-mail informado e a nova senha durante o fluxo de redefinição.

### 4.3. `useAuth.ts`

**Proteção de rotas**

- `ProtectedRoute.tsx` usa `isAuthenticated` do hook para bloquear o acesso de usuários não autenticados a páginas internas da aplicação.

**Cabeçalho e sessão**

- `Header.tsx` e `main.tsx` reutilizam o hook para exibir informações do usuário logado e restaurar a sessão ao recarregar a aplicação.

### 4.4. `auth.exceptions.ts`

**Registro, login e redefinição de senha (backend)**

- O `AuthController` delega toda a lógica de autenticação ao `AuthService`, que centraliza o lançamento das exceções customizadas. Isso garante que qualquer novo endpoint relacionado a autenticação (ex.: troca de e-mail, validação de conta) já tenha um vocabulário de erros pronto para reutilizar, sem precisar criar novas classes de exceção para casos já cobertos.

## 5. Hotspots e Frozen-spots na Arquitetura de Módulos

### 5.1. Aplicação dos Conceitos de Reutilização

A organização em camadas `apps/frontend` e `apps/backend`, cada uma com seus módulos centrais de infraestrutura, exemplifica os conceitos de **Frozen-spots** e **Hotspots** na reutilização de software, mesmo sem um pacote compartilhado do tipo Turborepo entre as camadas.

#### 5.1.1. Frozen-spots (Estrutura Invariante)

**1. Estrutura de Serviços do Frontend**

```
apps/frontend/src/
├── services/
│   ├── api.ts              // Frozen-spot: única instância Axios
│   ├── auth.service.ts     // consome api.ts
│   └── trails.service.ts   // consome api.ts
├── utils/
│   └── validators.ts       // Frozen-spot: regras de validação
└── hooks/
    └── useAuth.ts          // Frozen-spot: acesso único à authStore
```

**2. Padrão de Exceção do Backend**

```ts
// Frozen-spot: toda exceção de domínio estende uma exceção nativa do NestJS
export class AlgumaException extends BadRequestException {
  constructor() {
    super("Mensagem de erro padronizada");
  }
}
```

**3. Interceptors do Axios**

```ts
// Frozen-spot: todo serviço HTTP do frontend herda automaticamente
// (1) injeção de token JWT e (2) tratamento de sessão expirada (401)
api.interceptors.request.use(/* injeta token */);
api.interceptors.response.use(/* trata 401 */);
```

#### 5.1.2. Hotspots (Pontos de Variação)

**1. Endpoints consumidos via `api.ts`**

```ts
// Hotspot: cada serviço define suas próprias rotas
const response = await api.post('/auth/register', data);
const response = await api.get(`/trails/${id}`);
const response = await api.patch(`/progress/trails/${trailId}`, { ... });
```

**2. Regras específicas de validação**

```ts
// Hotspot: diferentes critérios podem ser adicionados a `validators`
isValidEmail: (email) => { /* regex de e-mail */ },
isStrongPassword: (password) => { /* regra mais rígida */ },
isWeakPassword: (password) => { /* regra mínima */ },
```

**3. Exceções específicas de cada fluxo**

```ts
// Hotspot: novas exceções podem ser adicionadas conforme o domínio cresce
export class EmailAlreadyExistsException extends ConflictException {
  /* ... */
}
export class InvalidTokenException extends UnauthorizedException {
  /* ... */
}
```

### 5.2. Framework Pattern em Escala de Aplicação

#### 5.2.1. Framework Core (Frozen-spots)

- **Cliente HTTP único**: `api.ts`, com interceptors fixos de autenticação.
- **Vocabulário de erros do domínio**: classes de exceção do backend, todas derivadas de exceções nativas do NestJS.
- **Ponto único de estado de autenticação**: `useAuth.ts` sobre `useAuthStore`.

#### 5.2.2. Application Points (Hotspots)

- **Endpoints e payloads específicos**: cada novo serviço (`auth.service.ts`, `trails.service.ts`, futuros serviços) define suas próprias rotas.
- **Regras de validação por formulário**: novos critérios podem ser adicionados a `validators.ts` sem alterar os formulários que já o utilizam.
- **Novas exceções de domínio**: podem ser criadas em `common/exceptions` seguindo o mesmo padrão, conforme novos fluxos (ex.: módulos, progresso) precisem de erros próprios.

### 5.3. Benefícios da Abordagem

#### 5.3.1. Reutilização Dentro da Camada

```ts
// auth.service.ts
import api from "./api";

// trails.service.ts
import api from "./api";

// futuro: notifications.service.ts
import api from "./api";
```

#### 5.3.2. Tratamento de Erro Consistente

```ts
// AuthService (backend)
if (existingUser) throw new EmailAlreadyExistsException(email);
if (senha.length < 8) throw new WeakPasswordException();
if (!user) throw new InvalidCredentialsException();
```

#### 5.3.3. Evolução Independente das Camadas

- Como frontend e backend não compartilham um pacote comum, cada camada pode evoluir seus módulos internos (`api.ts`, `auth.exceptions.ts`) de forma independente, desde que o contrato HTTP entre elas seja respeitado — uma característica importante para o time poder trabalhar em paralelo nas duas aplicações.

### 5.4. Comparação com a Abordagem de Pacotes Compartilhados (Turborepo)

| Aspecto                   | Módulos internos por camada (este projeto)             | Pacotes compartilhados via Turborepo                         |
| ------------------------- | ------------------------------------------------------ | ------------------------------------------------------------ |
| **Reutilização**          | Dentro da mesma camada (frontend ou backend)           | Cross-layer, via `packages/` importável por qualquer app     |
| **Acoplamento**           | Frontend e backend acoplados apenas pelo contrato HTTP | Apps acoplados também ao código do pacote compartilhado      |
| **Manutenção**            | Centralizada nos módulos frozen-spot de cada camada    | Centralizada em um único pacote para todas as camadas        |
| **Complexidade de setup** | Baixa (sem workspace/monorepo tooling)                 | Mais alta (configuração de workspaces, build, versionamento) |
| **Evolução**              | Hotspots bem definidos dentro de cada camada           | Hotspots compartilhados entre múltiplas aplicações           |

## 6. Senso Crítico e Trabalho em Equipe

A elaboração deste artefato exigiu **senso crítico** para identificar, dentro de um projeto que não adota um monorepo com pacotes compartilhados, quais módulos efetivamente cumprem o papel de reutilização — analisando onde a duplicação de código foi evitada por meio de uma instância única de cliente HTTP, de funções utilitárias de validação e de um vocabulário padronizado de exceções de domínio.

Foi igualmente importante o **trabalho em equipe**, com responsabilidades divididas entre quem atuou na codificação dos módulos do frontend e do backend e quem organizou e redigiu a documentação, de forma a tornar explícitos os pontos de reuso já existentes no código e facilitar que o time continue aplicando o mesmo padrão em novas funcionalidades.

## 7. Conclusão

Os módulos analisados — `api.ts`, `validators.ts` e `useAuth.ts` no frontend, e `auth.exceptions.ts` no backend — cumprem papéis centrais na arquitetura do projeto:

1. **Centralizar infraestrutura técnica reutilizável**, como a comunicação HTTP autenticada (`api.ts`) e o acesso ao estado global de autenticação (`useAuth.ts`), evitando que cada novo serviço ou componente reimplemente essa lógica.
2. **Padronizar regras de negócio recorrentes**, como validação de e-mail/senha (`validators.ts`) e o vocabulário de erros do domínio de autenticação (`auth.exceptions.ts`), garantindo consistência entre diferentes fluxos da aplicação.

Essa abordagem:

- Reduz duplicação de código dentro de cada camada;
- Facilita a manutenção e a evolução incremental do projeto;
- Cria uma base sólida para, no futuro, evoluir esses módulos internos para pacotes compartilhados (por exemplo, via Turborepo), caso o projeto passe a ter mais aplicações que precisem reaproveitar a mesma lógica entre frontend e backend.
